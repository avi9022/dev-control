# AI Automation Kanban — Design Document

## Overview

A developer AI automation system for DevControl. Users add tasks to a kanban board. Agent phases (powered by Claude Code CLI) automatically plan, implement, and review the work. The system is designed as a configurable pipeline of phases, where each phase is either manual (waits for user) or agent-powered (spawns Claude Code).

## Architecture

```
Renderer Process (React)
├── Top-level tab toggle: DevControl <-> AI Kanban
├── Kanban board view (columns by phase)
├── Task detail view (terminal + plan + diff/review + history)
└── AI Automation settings (knowledge docs, phase prompts, rules)

Main Process (Electron)
├── src/electron/ai-automation/
│   ├── task-manager.ts       # Task state machine, storage, queue
│   ├── agent-runner.ts       # Spawn claude CLI, pipe I/O, concurrency
│   ├── prompt-builder.ts     # Assemble prompts per phase (global + phase + project + task)
│   ├── worktree-manager.ts   # Git worktree/branch creation & cleanup
│   └── diff-parser.ts        # Parse git diffs for review UI
```

## Task Lifecycle (State Machine)

```
BACKLOG → TODO → PLANNING → IN_PROGRESS ⇄ AGENT_REVIEW → HUMAN_REVIEW → DONE
```

Transitions:
- BACKLOG → TODO: **Manual** (user drags/clicks)
- TODO → PLANNING: **Automatic** (system picks up task)
- PLANNING → IN_PROGRESS: **Automatic** (plan completes)
- IN_PROGRESS → AGENT_REVIEW: **Automatic** (worker completes)
- AGENT_REVIEW → IN_PROGRESS: **Automatic** (reviewer rejects, if cycles < max)
- AGENT_REVIEW → HUMAN_REVIEW: **Automatic** (reviewer approves OR max cycles reached)
- HUMAN_REVIEW → DONE: **Manual** (user approves)

## Agent System

### Prompt Assembly (prompt-builder.ts)

Four layers merged in order:
1. Global rules (user-defined in DevControl settings)
2. Phase-specific system prompt (user-customizable per phase)
3. Per-project rules (CLAUDE.md etc. — auto-discovered from working directory)
4. Task context (description, plan, review comments, conversation history)

### Agent Roles

**Planner** — Receives task + knowledge docs. Explores codebases, asks clarifying questions, produces a structured plan.

**Worker** — Receives task + plan. Implements changes in worktree/branch. Creates commits.

**Reviewer** — Receives task + plan + diff. Reviews code against plan and rules. Approves or rejects with comments.

### Project Context via Knowledge Docs

No project registry. Instead, users create knowledge documents (markdown) describing their system architecture, project relationships, conventions. An auto-generate feature lets users point to project directories and spawns a Claude Code agent to explore and produce these docs.

### Process Management (agent-runner.ts)

- Spawns `claude` CLI with `--system-prompt`, `--message`, `--output-format stream-json`
- stdout piped to xterm.js terminal in UI
- stdin piped from terminal for user responses
- JSON stream parsed to detect "agent needs input" events
- PID tracked for cancellation (uses tree-kill)
- Concurrency queue: tasks wait if at max concurrency limit

## UI Design

### Kanban Board View
- 7 columns matching phases
- Task cards: title, phase badge, notification indicator, review cycle count
- Drag between Backlog <-> Todo only
- Top bar: New Task button, concurrency indicator, settings link

### Task Detail View
- Header: title, phase, git info, controls (pause/cancel/restart)
- Tabs:
  - **Terminal**: xterm.js with Claude Code output. Notification banner when agent waits for input.
  - **Plan**: rendered markdown of planner output
  - **Diff/Review**: file tree + side-by-side or unified diff + inline comments (agent + human)
  - **History**: phase transition timeline

### Settings View
- Knowledge Docs: CRUD + auto-generate
- Phase Prompts: text editors per phase
- Global Rules: markdown editor
- General: concurrency, review cycles, git strategy defaults

## Human Review

- Diff generated via `git diff main...<branch>`
- Parsed into files → hunks → lines
- File tree sidebar with change counts
- Inline commenting on diff lines
- Agent review comments displayed inline
- Approve → DONE, Request Changes → IN_PROGRESS with comments serialized as context

## Future Enhancements

- **Customizable pipeline**: users add/remove/reorder phases, configure each as manual or agent
- **Claude API direct mode**: alternative to CLI for more structured control
- **Structured output parsing**: use `stream-json` format to build richer UI (progress indicators, file change tracking)
- **Task templates**: predefined task types with pre-filled prompts (bug fix, feature, refactor)
- **Task dependencies**: tasks that block other tasks
- **Auto-generate knowledge docs on schedule**: periodically re-explore projects to keep docs fresh
- **Integration with GitHub**: auto-create PRs, sync review comments
- **Agent memory**: persistent learnings across tasks (what worked, what didn't)
- **Cost tracking**: track API/token usage per task
- **Notifications**: system notifications when tasks need attention (not just in-app badges)
